# TurboModule Deep Dive — iOS

TurboModules são a substituição da Nova Arquitetura para a bridge legada `NativeModules`. No iOS, a implementação abrange quatro camadas: uma spec TypeScript que funciona como contrato, o Codegen que traduz esse contrato em interfaces nativas em tempo de build, um arquivo Objective-C++ que faz a ponte entre JSI e seu código, e uma classe Swift que contém a lógica de negócio real. Este documento cobre cada camada em profundidade, depois explica como `RCTTurboModuleManager` e seu delegate se encaixam, como o carregamento é controlado, e como testar e migrar de forma incremental.

## 1. Arquitetura: da Spec TypeScript ao Swift

O caminho de uma chamada JavaScript até uma função Swift é determinístico e totalmente verificado em tempo de build.

```
specs/NativePayment.ts          ← você escreve: o contrato
         │
         ▼  Codegen (roda no pod install)
build/generated/ios/
  NativePaymentSpec.h           ← protocolo ObjC++ + classe JSI C++
  NativePaymentSpec-generated.mm ← tabela de dispatch JSI (nunca edite)
         │
         ▼  você escreve
RCTNativePayment.mm             ← glue ObjC++ que adota o protocolo
PaymentHelper.swift             ← lógica de negócio Swift (@objcMembers)
         │
         ▼  runtime
JS: NativePayment.charge(amount) → JSI → RCTNativePayment.mm → PaymentHelper.swift
```

A spec é a única fonte de verdade. Se um método está na spec mas ausente da implementação `.mm`, o build falha com um erro de conformidade de protocolo. Não há descoberta em tempo de execução, sem correspondência de strings, sem parsing de JSON — tudo é verificado em tempo de compilação.

### Spec TypeScript

```ts
// specs/NativePayment.ts
import type { TurboModule } from 'react-native';
import { TurboModuleRegistry } from 'react-native';

export interface Spec extends TurboModule {
  charge(amount: number, currency: string): Promise<string>;
  getLastTransactionId(): string | null;
  cancelPending(): void;
}

export default TurboModuleRegistry.strictGet<Spec>('NativePayment');
```

`strictGet` (equivalente a `getEnforcing`) lança imediatamente se o módulo não estiver registrado — adequado para um módulo que deve existir. Use `get` para módulos opcionais.

### Saída do Codegen (Simplificada)

Após `pod install`, o protocolo ObjC++ gerado em `build/generated/ios/` se parece com:

```objc
// AUTO-GENERATED — do not edit
@protocol NativePaymentSpec <RCTBridgeModule, RCTTurboModule>
- (void)charge:(double)amount
      currency:(NSString *)currency
        resolve:(RCTPromiseResolveBlock)resolve
         reject:(RCTPromiseRejectBlock)reject;
- (NSString *)getLastTransactionId;
- (void)cancelPending;
@end
```

### Helper Swift

Swift não pode implementar um TurboModule diretamente porque JSI é C++ e Swift não possui interoperabilidade C++ bruta compatível com a convenção de chamada do JSI. Todo ponto de entrada de TurboModule deve ser `.mm`. Seu código Swift fica em uma classe helper exposta ao Objective-C via `@objcMembers`.

```swift
// PaymentHelper.swift
import Foundation

@objcMembers
class PaymentHelper: NSObject {
  private var lastTransactionId: String?
  private var pendingTask: URLSessionDataTask?

  func charge(amount: Double, currency: String) async throws -> String {
    let body: [String: Any] = ["amount": amount, "currency": currency]
    let data = try await PaymentNetwork.post(path: "/charge", body: body)
    let decoded = try JSONDecoder().decode(ChargeResponse.self, from: data)
    lastTransactionId = decoded.transactionId
    return decoded.transactionId
  }

  func getLastTransactionId() -> String? {
    return lastTransactionId
  }

  func cancelPending() {
    pendingTask?.cancel()
    pendingTask = nil
  }
}
```

### Glue ObjC++

```objc
// RCTNativePayment.mm
#import <React/RCTBridgeModule.h>
#import <ReactCommon/RCTTurboModule.h>
#import "MyApp-Swift.h"   // exposes PaymentHelper to ObjC

// Generated by Codegen:
#import <AppSpecs/AppSpecsJSI.h>

using namespace facebook::react;

@interface RCTNativePayment : NSObject <RCTBridgeModule, RCTTurboModule, NativePaymentSpec>
@end

@implementation RCTNativePayment {
  PaymentHelper *_helper;
}

RCT_EXPORT_MODULE(NativePayment)

- (instancetype)init {
  if (self = [super init]) {
    _helper = [[PaymentHelper alloc] init];
  }
  return self;
}

- (void)charge:(double)amount
      currency:(NSString *)currency
        resolve:(RCTPromiseResolveBlock)resolve
         reject:(RCTPromiseRejectBlock)reject {
  [_helper chargeWithAmount:amount currency:currency completionHandler:^(NSString *txId, NSError *error) {
    if (error) {
      reject(@"CHARGE_FAILED", error.localizedDescription, error);
    } else {
      resolve(txId);
    }
  }];
}

- (NSString *)getLastTransactionId {
  return [_helper getLastTransactionId];
}

- (void)cancelPending {
  [_helper cancelPending];
}

- (std::shared_ptr<TurboModule>)getTurboModule:(const ObjCTurboModule::InitParams &)params {
  return std::make_shared<NativePaymentSpecJSI>(params);
}

@end
```

O argumento string para `RCT_EXPORT_MODULE` — `NativePayment` — deve corresponder exatamente ao nome usado em `TurboModuleRegistry.strictGet('NativePayment')`. Esta é a fonte mais comum de erros de "module not found" em tempo de execução.

## 2. RCTTurboModuleManager

`RCTTurboModuleManager` é o objeto iOS responsável por resolver nomes de módulos em instâncias de TurboModule. Quando JavaScript chama `TurboModuleRegistry.get('NativePayment')`, o JSI encaminha a busca ao manager, que consulta seu delegate para encontrar ou criar o módulo.

O manager reside no `AppDelegate`. Ele é criado durante a inicialização do app, mantém uma referência pelo tempo de vida da surface React, e é destruído quando a bridge ou o host são encerrados.

## 3. RCTTurboModuleManagerDelegate no AppDelegate.mm

O `AppDelegate` deve estar em conformidade com `RCTTurboModuleManagerDelegate`. O protocolo possui três métodos que juntos cobrem todas as estratégias de resolução.

```objc
// AppDelegate.mm — full delegate implementation
#import "AppDelegate.h"
#import "RCTNativePayment.h"

#import <React/RCTAppSetupUtils.h>
#import <ReactCommon/RCTTurboModuleManager.h>
#import <React/CoreModulesPlugins.h>

@interface AppDelegate () <RCTTurboModuleManagerDelegate>
@end

@implementation AppDelegate

// Chamado para cada nome de módulo que o JS solicita.
// Retorne uma Class para deixar o manager instanciá-la via getModuleInstanceFromClass:.
// Retorne nil para pular a resolução baseada em classe e cair em getTurboModule:.
- (Class)getModuleClassFromName:(const char *)name {
  return RCTCoreModulesClassProvider(name);
}

// Chamado após getModuleClassFromName: quando você precisa injetar dependências
// no módulo no momento da instanciação (ex.: um objeto de serviço compartilhado).
// Retorne nil para usar o caminho padrão [[moduleClass alloc] init].
- (id<RCTTurboModule>)getModuleInstanceFromClass:(Class)moduleClass {
  if (moduleClass == RCTNativePayment.class) {
    // Injeta um PaymentService compartilhado, por exemplo:
    return [[RCTNativePayment alloc] initWithService:self.sharedPaymentService];
  }
  // Para todos os outros módulos, use o inicializador padrão.
  return RCTAppSetupDefaultModuleFromClass(moduleClass);
}

// Chamado para módulos que fornecem um TurboModule C++ diretamente
// (ou seja, um módulo que expõe um jsi::HostObject, não apenas um wrapper ObjC).
// Retorne nullptr para usar o caminho de resolução padrão baseado em ObjC.
- (std::shared_ptr<facebook::react::TurboModule>)
    getTurboModule:(const std::string &)name
         jsInvoker:(std::shared_ptr<facebook::react::CallInvoker>)jsInvoker {
  return nullptr;
}

@end
```

Os três métodos formam uma cadeia de resolução: `getModuleClassFromName:` é consultado primeiro; se retornar uma classe, `getModuleInstanceFromClass:` é chamado para produzir uma instância; `getTurboModule:jsInvoker:` é o fallback para módulos C++ puros sem classe ObjC.

## 4. Inicialização do RCTTurboModuleManager: Bridge vs Bridgeless

### Modo Bridge (RN 0.73 e abaixo, ou apps que não optaram pelo bridgeless)

```objc
// AppDelegate.mm — bridge mode
- (BOOL)application:(UIApplication *)application
    didFinishLaunchingWithOptions:(NSDictionary *)launchOptions {

  RCTAppSetupPrepareApp(application, YES); // YES = TurboModules habilitado

  _bridge = [[RCTBridge alloc] initWithDelegate:self launchOptions:launchOptions];

  _turboModuleManager = [[RCTTurboModuleManager alloc]
      initWithBridge:_bridge
            delegate:self
           jsInvoker:_bridge.jsCallInvoker];

  [_bridge setRCTTurboModuleRegistry:_turboModuleManager];

  // ... cria RCTRootView e window
  return YES;
}
```

### Modo Bridgeless (padrão no RN 0.74+, Expo SDK 52+)

No modo bridgeless, `RCTHost` substitui `RCTBridge`. O TurboModule manager é criado internamente pelo `RCTHost` e o delegate é passado durante a inicialização do host:

```objc
// AppDelegate.mm — bridgeless mode
- (BOOL)application:(UIApplication *)application
    didFinishLaunchingWithOptions:(NSDictionary *)launchOptions {

  _reactHost = [[RCTHost alloc]
      initWithBundleURLProvider:^NSURL *{ return [self bundleURL]; }
        hostControllerDelegate:self
            turboModuleManager:self   // self conforms to RCTTurboModuleManagerDelegate
                  jsEngineProvider:self
                     launchOptions:launchOptions];

  [_reactHost start];
  return YES;
}
```

Em ambos os modos, o protocolo de delegate é idêntico. A diferença está apenas em quem possui e inicializa a instância de `RCTTurboModuleManager`.

## 5. Carregamento de Módulos: Eager vs Lazy

TurboModules são lazy por padrão. A classe do módulo é registrada em tempo de build através de `RCT_EXPORT_MODULE`, mas o objeto não é alocado até que o JavaScript chame pela primeira vez `TurboModuleRegistry.get('ModuleName')`.

O carregamento lazy mantém o tempo de inicialização baixo: um módulo que lida com sincronização em background nunca é instanciado em uma sessão onde o usuário apenas lê conteúdo em cache.

### Forçando Inicialização Eager

Para forçar um módulo a inicializar antes de qualquer chamada JS, implemente `requiresMainQueueSetup` e retorne `YES`:

```objc
+ (BOOL)requiresMainQueueSetup {
  return YES;  // Instanciado na main queue antes do primeiro frame JS
}
```

Isso é adequado para módulos que precisam configurar estado global (como um SDK de logging) antes de qualquer código ser executado. Módulos que retornam `YES` são instanciados sincronamente durante a inicialização da bridge ou do host, o que aumenta o custo de startup — use com moderação.

### RCTTurboModuleSharedNothing

O protocolo `RCTTurboModuleSharedNothing` marca um módulo como não tendo estado mutável compartilhado entre instâncias. Quando adotado, o runtime pode criar instâncias por chamador ao invés de um singleton, o que habilita isolamento mais rígido em ambientes de teste e permite ao runtime adiar ainda mais a alocação:

```objc
@interface RCTNativePayment : NSObject <RCTBridgeModule, RCTTurboModule, RCTTurboModuleSharedNothing>
@end
```

Em apps de produção, este protocolo raramente é necessário, mas é útil ao escrever módulos que devem se comportar de forma idêntica tanto quando chamados da surface principal quanto de uma `RCTFabricSurface` secundária.

## 6. TurboModules Assíncronos: RCTTurboModuleWithJSIBindings

Alguns módulos precisam instalar funções ou objetos personalizados diretamente no runtime JS ao invés de expor métodos através do protocolo Codegen. Exemplos incluem relatores de crash que interceptam exceções JS, substituições customizadas de `console`, e ferramentas de rastreamento de performance.

Esses módulos adotam `RCTTurboModuleWithJSIBindings`:

```objc
// RCTPerformanceTracer.h
@interface RCTPerformanceTracer : NSObject
    <RCTBridgeModule, RCTTurboModule, RCTTurboModuleWithJSIBindings>
@end

// RCTPerformanceTracer.mm
RCT_EXPORT_MODULE(NativePerformanceTracer)

- (void)installJSIBindingsWithRuntime:(facebook::jsi::Runtime &)runtime {
  using namespace facebook::jsi;

  // Instala uma função global __perfMark chamável a partir do JS
  auto markFn = Function::createFromHostFunction(
    runtime,
    PropNameID::forAscii(runtime, "__perfMark"),
    1,
    [](Runtime &rt, const Value &, const Value *args, size_t count) -> Value {
      if (count > 0 && args[0].isString()) {
        std::string label = args[0].getString(rt).utf8(rt);
        // Registra o mark no buffer de trace nativo
        PerformanceTraceBuffer::shared().mark(label);
      }
      return Value::undefined();
    }
  );
  runtime.global().setProperty(runtime, "__perfMark", std::move(markFn));
}

- (std::shared_ptr<facebook::react::TurboModule>)
    getTurboModule:(const facebook::react::ObjCTurboModule::InitParams &)params {
  return std::make_shared<facebook::react::NativePerformanceTracerSpecJSI>(params);
}
```

`installJSIBindingsWithRuntime:` é chamado uma vez após o runtime JS ser inicializado, antes que o bundle JS seja executado. A função que você instala está disponível para o bundle imediatamente.

## 7. TurboModules Síncronos

Métodos em um TurboModule que retornam um valor diretamente (não via Promise) executam sincronamente na thread JS. Não há fila assíncrona, sem serialização — o JSI chama a tabela de dispatch C++ e o valor de retorno flui de volta para o JS no mesmo stack frame.

```ts
// Na spec: tipo de retorno síncrono
getLastTransactionId(): string | null;
```

```objc
// No .mm: método ObjC chamado sincronamente a partir do JS
- (NSString *)getLastTransactionId {
  return [_helper getLastTransactionId];
}
```

Para módulos que precisam retornar um `jsi::HostObject` completamente customizado ao invés de um dos tipos de retorno suportados pelo Codegen, sobrescreva `getTurboModule:jsInvoker:` no delegate do AppDelegate para retornar o objeto C++ diretamente:

```objc
- (std::shared_ptr<facebook::react::TurboModule>)
    getTurboModule:(const std::string &)name
         jsInvoker:(std::shared_ptr<facebook::react::CallInvoker>)jsInvoker {
  if (name == "NativeCustomHost") {
    return std::make_shared<CustomHostObject>(jsInvoker);
  }
  return nullptr;
}
```

`CustomHostObject` estende `facebook::jsi::HostObject` e sobrescreve `get` e `set` para interceptar qualquer acesso a propriedades a partir do JS. Este padrão é usado por bibliotecas baseadas em JSI como MMKV e Reanimated.

## 8. Testes

### Testando a Implementação Swift com XCTest

A classe helper Swift pode ser testada independentemente do React Native. Sem JSI, sem glue ObjC++, sem Codegen:

```swift
// PaymentHelperTests.swift
import XCTest
@testable import MyApp

final class PaymentHelperTests: XCTestCase {
  var helper: PaymentHelper!

  override func setUp() {
    super.setUp()
    helper = PaymentHelper()
  }

  func testGetLastTransactionIdReturnsNilInitially() {
    XCTAssertNil(helper.getLastTransactionId())
  }

  func testCancelPendingDoesNotThrow() {
    // Deve ser um no-op quando nada está pendente
    XCTAssertNoThrow(helper.cancelPending())
  }

  func testChargeStoresTransactionId() async throws {
    // Usando um stub URLProtocol para isolamento de rede
    URLProtocol.registerClass(MockPaymentProtocol.self)
    defer { URLProtocol.unregisterClass(MockPaymentProtocol.self) }

    MockPaymentProtocol.responseData = #"{"transactionId":"tx_abc123"}"#.data(using: .utf8)

    let txId = try await helper.charge(amount: 9.99, currency: "USD")

    XCTAssertEqual(txId, "tx_abc123")
    XCTAssertEqual(helper.getLastTransactionId(), "tx_abc123")
  }
}
```

Esta é a principal superfície de testes para TurboModules iOS — a lógica de negócio pertence ao Swift, onde o XCTest pode alcançá-la sem a necessidade de configuração do runtime nativo.

### Mockando o Lado JS com Jest

No lado JavaScript, `TurboModuleRegistry.strictGet` falha no Jest porque não há runtime nativo. Mocke o módulo spec antes de qualquer import que o utilize:

```ts
// __mocks__/NativePayment.ts  (coloque ao lado do arquivo spec)
const NativePaymentMock = {
  charge: jest.fn().mockResolvedValue('tx_test_001'),
  getLastTransactionId: jest.fn().mockReturnValue('tx_test_001'),
  cancelPending: jest.fn(),
};

export default NativePaymentMock;
```

No seu teste:

```ts
// payment.test.ts
jest.mock('../specs/NativePayment');
import NativePayment from '../specs/NativePayment';
import { processPayment } from '../services/PaymentService';

describe('PaymentService', () => {
  beforeEach(() => jest.clearAllMocks());

  it('calls charge with correct arguments', async () => {
    await processPayment(19.99, 'BRL');
    expect(NativePayment.charge).toHaveBeenCalledWith(19.99, 'BRL');
  });

  it('returns the transaction id', async () => {
    const result = await processPayment(19.99, 'BRL');
    expect(result.transactionId).toBe('tx_test_001');
  });
});
```

## 9. Migração a partir dos NativeModules Legados

Módulos que usam o protocolo legado `RCTBridgeModule` continuam funcionando no RN 0.76+ através da camada de interoperabilidade de TurboModule, que os envolve automaticamente em uma facade de TurboModule. Você pode migrar de forma incremental sem quebrar o app.

### Funcionamento em Paralelo Durante a Migração

Mantenha as declarações legadas `RCT_EXPORT_MODULE` e `RCT_EXPORT_METHOD` enquanto adiciona a conformidade com TurboModule:

```objc
// RCTNativePayment.mm — durante a migração
@interface RCTNativePayment : NSObject
    <RCTBridgeModule,    // legado — mantém chamadores existentes funcionando
     RCTTurboModule,     // novo — habilita resolução TurboModule
     NativePaymentSpec>  // novo — protocolo gerado pelo Codegen
@end

@implementation RCTNativePayment

RCT_EXPORT_MODULE(NativePayment)

// Export legado — usado pela camada de interop até a spec estar disponível
RCT_EXPORT_METHOD(charge:(double)amount
                  currency:(NSString *)currency
                  resolver:(RCTPromiseResolveBlock)resolve
                  rejecter:(RCTPromiseRejectBlock)reject) {
  [self charge:amount currency:currency resolve:resolve reject:reject];
}

// Novo método TurboModule — usado pelo JS assim que a spec estiver conectada
- (void)charge:(double)amount
      currency:(NSString *)currency
        resolve:(RCTPromiseResolveBlock)resolve
         reject:(RCTPromiseRejectBlock)reject {
  // implementação compartilhada
}

- (std::shared_ptr<facebook::react::TurboModule>)
    getTurboModule:(const facebook::react::ObjCTurboModule::InitParams &)params {
  return std::make_shared<facebook::react::NativePaymentSpecJSI>(params);
}

@end
```

Assim que todos os pontos de chamada usarem a spec tipada e a camada TypeScript não usar mais `NativeModules.NativePayment`, remova as declarações `RCT_EXPORT_METHOD` e a conformidade legada com `RCTBridgeModule`.

### Ressalvas da Camada de Interoperabilidade

A camada de interop cobre a maioria dos tipos de método, mas possui lacunas conhecidas: callbacks (não Promises), métodos que aceitam `RCTResponseSenderBlock`, e módulos que sobrescrevem `methodQueue` para executar em uma fila serial customizada podem se comportar de forma diferente sob a camada de interop. Teste cada módulo migrado individualmente antes de remover o export legado.

## 10. Armadilhas Comuns

### Esquecer o RCT_EXPORT_MODULE

Sem `RCT_EXPORT_MODULE(NativePayment)`, o runtime Objective-C nunca registra a classe no registro de módulos do React Native. O `getModuleClassFromName:` do delegate do `TurboModuleManager` não a encontrará, e o JavaScript recebe um erro "module not found" em tempo de execução, não em tempo de build.

A macro deve aparecer dentro de `@implementation`, não dentro de `@interface`. Colocá-la no header causa um erro de linker.

### Incompatibilidade de Nome de Módulo Entre a Spec e o .mm

A string em `RCT_EXPORT_MODULE(NativePayment)` deve corresponder exatamente à string em `TurboModuleRegistry.strictGet('NativePayment')`, incluindo maiúsculas e minúsculas. Um erro comum é usar `RCT_EXPORT_MODULE()` sem argumento, que usa como padrão o nome da classe (ex.: `RCTNativePayment`) — que então não corresponde a `'NativePayment'` na spec.

Sempre passe o nome do módulo explicitamente:

```objc
RCT_EXPORT_MODULE(NativePayment)   // corresponds to TurboModuleRegistry.strictGet('NativePayment')
```

### Pod Não Adicionado ao Podfile

Se o módulo é distribuído como um pod separado e não está listado no `Podfile` do app consumidor, o Codegen nunca roda para ele, os headers gerados ficam ausentes, e o build do Xcode falha com "file not found" no import do header de spec gerado. Todo pod que contém um TurboModule deve aparecer no `Podfile` antes do `pod install`:

```ruby
# Podfile
pod 'NativePaymentModule', path: '../modules/native-payment'
```

Execute novamente `bundle exec pod install` após adicionar a entrada.

### pod_target_xcconfig Ausente para o Padrão C++

Os arquivos `.mm` gerados utilizam funcionalidades do C++17. Sem a flag de compilador correta, o build falha com erros obscuros de template:

```ruby
# No seu podspec:
s.pod_target_xcconfig = {
  "CLANG_CXX_LANGUAGE_STANDARD" => "c++17",
  "HEADER_SEARCH_PATHS" => "$(PODS_ROOT)/boost"
}
```

### Nome Codegen Incorreto no package.json

O campo `name` em `codegenConfig` torna-se o nome do framework iOS e o prefixo dos arquivos gerados. Se esse nome não corresponder ao import no seu arquivo `.mm`, o build falha:

```objc
// Se codegenConfig.name for "AppSpecs":
#import <AppSpecs/AppSpecsJSI.h>  // correto

// Se codegenConfig.name for "MyModuleSpecs":
#import <MyModuleSpecs/MyModuleSpecsJSI.h>  // deve corresponder
```

Execute `pod install` e inspecione `build/generated/ios/` para confirmar o nome real do diretório antes de escrever o import.

### getTurboModule: Retornando um Ponteiro Nulo

Se `getTurboModule:` retornar um `shared_ptr` nulo, o runtime silenciosamente cai de volta para o caminho de resolução de módulo legado (ou retorna undefined para o JS). Adicione um log dentro do método durante o desenvolvimento para confirmar que ele está sendo alcançado:

```objc
- (std::shared_ptr<facebook::react::TurboModule>)
    getTurboModule:(const facebook::react::ObjCTurboModule::InitParams &)params {
  NSLog(@"[RCTNativePayment] getTurboModule: called");
  return std::make_shared<facebook::react::NativePaymentSpecJSI>(params);
}
```

Se o log nunca aparecer, a classe não está sendo resolvida pelo delegate — verifique se `RCT_EXPORT_MODULE` está presente e se o nome do módulo corresponde.
