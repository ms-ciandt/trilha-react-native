## The Same Patterns You Already Use

Feature-based folder structure, custom hooks for data fetching, a global store for auth state, shared components — none of this is new. The architecture patterns that work in a large React web app work in React Native for the same reasons.

The additions are small: a `native/` layer to isolate the JS↔native boundary, and navigation as infrastructure rather than something owned by individual features.

---

## Suggested Project Structure

```
src/
  app/
    navigation/
      RootNavigator.tsx
      AppTabs.tsx
    store/
      authStore.ts
    config/
      env.ts
  features/
    auth/
      screens/
        LoginScreen.tsx
        RegisterScreen.tsx
      components/
      hooks/
      api/
    feed/
      screens/
      components/
      api/
    profile/
  shared/
    components/
    hooks/
    styles/
  native/
    modules/
    ui/
```

- `app/` — infrastructure: navigation tree, global stores, environment config. Nothing here knows about individual features.
- `features/` — one folder per product domain. Each owns its screens, hooks, API calls, and local components.
- `shared/` — reusable pieces that more than one feature needs. Keep the bar high; prefer a little duplication over pulling things into shared too early.
- `native/` — the bridge boundary: JS wrappers for native modules and native UI components. Isolating them here makes the surface between JS and native explicit and easy to audit.

## Domain Hooks

The pattern is identical to React web: a hook fetches data, manages loading state, and returns what the component needs. The component stays thin.

```ts
// src/features/feed/hooks/useFeed.ts
import { useEffect, useState } from 'react';
import { fetchFeed } from '../api/feedApi';

type FeedItem = { id: string; title: string };

export function useFeed() {
  const [items, setItems] = useState<FeedItem[]>([]);
  const [loading, setLoading] = useState(true);

  useEffect(() => {
    fetchFeed()
      .then(setItems)
      .finally(() => setLoading(false));
  }, []);

  return { items, loading };
}
```

```tsx
// src/features/feed/screens/FeedScreen.tsx
import React from 'react';
import { View, ActivityIndicator } from 'react-native';
import { useFeed } from '../hooks/useFeed';
import { FeedList } from '../components/FeedList';

export function FeedScreen() {
  const { items, loading } = useFeed();

  if (loading) return <ActivityIndicator />;

  return (
    <View>
      <FeedList items={items} />
    </View>
  );
}
```

`fetchFeed` in `features/feed/api/feedApi.ts` is a plain async function wrapping a `fetch` call. Nothing RN-specific — the same module could run in a web app.

## Global State with Zustand

```ts
// src/app/store/authStore.ts
import { create } from 'zustand';

interface AuthState {
  isAuthenticated: boolean;
  token?: string;
  login: (token: string) => void;
  logout: () => void;
}

export const useAuthStore = create<AuthState>((set) => ({
  isAuthenticated: false,
  token: undefined,
  login: (token) => set({ isAuthenticated: true, token }),
  logout: () => set({ isAuthenticated: false, token: undefined }),
}));
```

If you've used Zustand on the web, this is identical. If you've used Redux, Zustand is the same idea with far less boilerplate — a store is just a function that returns state and actions.

## Navigation as Infrastructure

Navigation (Stack/Tab/Drawer) belongs in `app/navigation`, not inside a feature. A feature that owns its own navigator becomes impossible to reorganize — its entry points are buried inside a product module.

```tsx
// src/app/navigation/RootNavigator.tsx
import React from 'react';
import { NavigationContainer } from '@react-navigation/native';
import { createBottomTabNavigator } from '@react-navigation/bottom-tabs';
import { FeedScreen } from '../../features/feed/screens/FeedScreen';
import { ProfileScreen } from '../../features/profile/screens/ProfileScreen';

const Tab = createBottomTabNavigator();

export function RootNavigator() {
  return (
    <NavigationContainer>
      <Tab.Navigator>
        <Tab.Screen name="Feed" component={FeedScreen} />
        <Tab.Screen name="Profile" component={ProfileScreen} />
      </Tab.Navigator>
    </NavigationContainer>
  );
}
```

Features navigate by calling `navigation.navigate('ScreenName')`. They know screen names, not how the navigator is assembled. This is the RN equivalent of keeping route definitions at the router level, not inside individual page components.
