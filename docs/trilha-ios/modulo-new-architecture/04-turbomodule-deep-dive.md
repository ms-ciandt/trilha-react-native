# TurboModule Deep Dive — iOS

TurboModules are the New Architecture replacement for the legacy `NativeModules` bridge. On iOS the implementation spans four layers: a TypeScript spec that acts as the contract, Codegen that translates that contract into native interfaces at build time, an Objective-C++ file that bridges JSI to your code, and a Swift class that holds the actual business logic. This document covers each layer in depth, then explains how `RCTTurboModuleManager` and its delegate fit together, how loading is controlled, and how to test and migrate incrementally.

## 1. Architecture: TypeScript Spec to Swift

The path from a JavaScript call to a Swift function is deterministic and entirely build-time-verified.

```
specs/NativePayment.ts          ← you write: the contract
         │
         ▼  Codegen (runs on pod install)
build/generated/ios/
  NativePaymentSpec.h           ← ObjC++ protocol + C++ JSI class
  NativePaymentSpec-generated.mm ← JSI dispatch table (never edit)
         │
         ▼  you write
RCTNativePayment.mm             ← ObjC++ glue that adopts the protocol
PaymentHelper.swift             ← Swift business logic (@objcMembers)
         │
         ▼  runtime
JS: NativePayment.charge(amount) → JSI → RCTNativePayment.mm → PaymentHelper.swift
```

The spec is the only source of truth. If a method is in the spec but missing from the `.mm` implementation, the build fails with a protocol conformance error. There is no runtime discovery, no string matching, no JSON parsing — everything is verified at compile time.

### TypeScript Spec

```ts
// specs/NativePayment.ts
import type { TurboModule } from 'react-native';
import { TurboModuleRegistry } from 'react-native';

export interface Spec extends TurboModule {
  charge(amount: number, currency: string): Promise<string>;
  getLastTransactionId(): string | null;
  cancelPending(): void;
}

export default TurboModuleRegistry.strictGet<Spec>('NativePayment');
```

`strictGet` (equivalent to `getEnforcing`) throws immediately if the module is not registered — appropriate for a module that must exist. Use `get` for optional modules.

### Codegen Output (Simplified)

After `pod install`, the generated ObjC++ protocol at `build/generated/ios/` looks like:

```objc
// AUTO-GENERATED — do not edit
@protocol NativePaymentSpec <RCTBridgeModule, RCTTurboModule>
- (void)charge:(double)amount
      currency:(NSString *)currency
        resolve:(RCTPromiseResolveBlock)resolve
         reject:(RCTPromiseRejectBlock)reject;
- (NSString *)getLastTransactionId;
- (void)cancelPending;
@end
```

### Swift Helper

Swift cannot implement a TurboModule directly because JSI is C++ and Swift has no raw C++ interop compatible with JSI's calling convention. Every TurboModule entry point must be `.mm`. Your Swift code lives in a helper class exposed to Objective-C via `@objcMembers`.

```swift
// PaymentHelper.swift
import Foundation

@objcMembers
class PaymentHelper: NSObject {
  private var lastTransactionId: String?
  private var pendingTask: URLSessionDataTask?

  func charge(amount: Double, currency: String) async throws -> String {
    let body: [String: Any] = ["amount": amount, "currency": currency]
    let data = try await PaymentNetwork.post(path: "/charge", body: body)
    let decoded = try JSONDecoder().decode(ChargeResponse.self, from: data)
    lastTransactionId = decoded.transactionId
    return decoded.transactionId
  }

  func getLastTransactionId() -> String? {
    return lastTransactionId
  }

  func cancelPending() {
    pendingTask?.cancel()
    pendingTask = nil
  }
}
```

### ObjC++ Glue

```objc
// RCTNativePayment.mm
#import <React/RCTBridgeModule.h>
#import <ReactCommon/RCTTurboModule.h>
#import "MyApp-Swift.h"   // exposes PaymentHelper to ObjC

// Generated by Codegen:
#import <AppSpecs/AppSpecsJSI.h>

using namespace facebook::react;

@interface RCTNativePayment : NSObject <RCTBridgeModule, RCTTurboModule, NativePaymentSpec>
@end

@implementation RCTNativePayment {
  PaymentHelper *_helper;
}

RCT_EXPORT_MODULE(NativePayment)

- (instancetype)init {
  if (self = [super init]) {
    _helper = [[PaymentHelper alloc] init];
  }
  return self;
}

- (void)charge:(double)amount
      currency:(NSString *)currency
        resolve:(RCTPromiseResolveBlock)resolve
         reject:(RCTPromiseRejectBlock)reject {
  [_helper chargeWithAmount:amount currency:currency completionHandler:^(NSString *txId, NSError *error) {
    if (error) {
      reject(@"CHARGE_FAILED", error.localizedDescription, error);
    } else {
      resolve(txId);
    }
  }];
}

- (NSString *)getLastTransactionId {
  return [_helper getLastTransactionId];
}

- (void)cancelPending {
  [_helper cancelPending];
}

- (std::shared_ptr<TurboModule>)getTurboModule:(const ObjCTurboModule::InitParams &)params {
  return std::make_shared<NativePaymentSpecJSI>(params);
}

@end
```

The string argument to `RCT_EXPORT_MODULE` — `NativePayment` — must match the name used in `TurboModuleRegistry.strictGet('NativePayment')` exactly. This is the most common source of runtime "module not found" errors.

## 2. RCTTurboModuleManager

`RCTTurboModuleManager` is the iOS object responsible for resolving module names to TurboModule instances. When JavaScript calls `TurboModuleRegistry.get('NativePayment')`, JSI routes the lookup to the manager, which consults its delegate to find or create the module.

The manager lives in `AppDelegate`. It is created during app startup, holds a reference for the lifetime of the React surface, and is destroyed when the bridge or host tears down.

## 3. RCTTurboModuleManagerDelegate in AppDelegate.mm

`AppDelegate` must conform to `RCTTurboModuleManagerDelegate`. The protocol has three methods that together cover all resolution strategies.

```objc
// AppDelegate.mm — full delegate implementation
#import "AppDelegate.h"
#import "RCTNativePayment.h"

#import <React/RCTAppSetupUtils.h>
#import <ReactCommon/RCTTurboModuleManager.h>
#import <React/CoreModulesPlugins.h>

@interface AppDelegate () <RCTTurboModuleManagerDelegate>
@end

@implementation AppDelegate

// Called for every module name that JS requests.
// Return a Class to let the manager instantiate it via getModuleInstanceFromClass:.
// Return nil to skip class-based resolution and fall through to getTurboModule:.
- (Class)getModuleClassFromName:(const char *)name {
  return RCTCoreModulesClassProvider(name);
}

// Called after getModuleClassFromName: when you need to inject dependencies
// into the module at instantiation time (e.g., a shared service object).
// Return nil to use the default [[moduleClass alloc] init] path.
- (id<RCTTurboModule>)getModuleInstanceFromClass:(Class)moduleClass {
  if (moduleClass == RCTNativePayment.class) {
    // Inject a shared PaymentService, for example:
    return [[RCTNativePayment alloc] initWithService:self.sharedPaymentService];
  }
  // For all other modules, use the default initializer.
  return RCTAppSetupDefaultModuleFromClass(moduleClass);
}

// Called for modules that provide a C++ TurboModule directly
// (i.e., a module that exposes a jsi::HostObject, not just an ObjC wrapper).
// Return nullptr to use the default ObjC-based resolution path.
- (std::shared_ptr<facebook::react::TurboModule>)
    getTurboModule:(const std::string &)name
         jsInvoker:(std::shared_ptr<facebook::react::CallInvoker>)jsInvoker {
  return nullptr;
}

@end
```

The three methods form a resolution chain: `getModuleClassFromName:` is consulted first; if it returns a class, `getModuleInstanceFromClass:` is called to produce an instance; `getTurboModule:jsInvoker:` is the fallback for pure C++ modules with no ObjC class.

## 4. RCTTurboModuleManager Initialization: Bridge vs Bridgeless

### Bridge Mode (RN 0.73 and below, or apps that have not opted into bridgeless)

```objc
// AppDelegate.mm — bridge mode
- (BOOL)application:(UIApplication *)application
    didFinishLaunchingWithOptions:(NSDictionary *)launchOptions {

  RCTAppSetupPrepareApp(application, YES); // YES = TurboModules enabled

  _bridge = [[RCTBridge alloc] initWithDelegate:self launchOptions:launchOptions];

  _turboModuleManager = [[RCTTurboModuleManager alloc]
      initWithBridge:_bridge
            delegate:self
           jsInvoker:_bridge.jsCallInvoker];

  [_bridge setRCTTurboModuleRegistry:_turboModuleManager];

  // ... create RCTRootView and window
  return YES;
}
```

### Bridgeless Mode (RN 0.74+ default, Expo SDK 52+)

In bridgeless mode, `RCTHost` replaces `RCTBridge`. The TurboModule manager is created internally by `RCTHost` and the delegate is passed during host initialization:

```objc
// AppDelegate.mm — bridgeless mode
- (BOOL)application:(UIApplication *)application
    didFinishLaunchingWithOptions:(NSDictionary *)launchOptions {

  _reactHost = [[RCTHost alloc]
      initWithBundleURLProvider:^NSURL *{ return [self bundleURL]; }
        hostControllerDelegate:self
            turboModuleManager:self   // self conforms to RCTTurboModuleManagerDelegate
                  jsEngineProvider:self
                     launchOptions:launchOptions];

  [_reactHost start];
  return YES;
}
```

In both modes the delegate protocol is identical. The difference is only in who owns and initializes the `RCTTurboModuleManager` instance.

## 5. Module Loading: Eager vs Lazy

TurboModules are lazy by default. The module class is registered at build time through `RCT_EXPORT_MODULE`, but the object is not allocated until JavaScript first calls `TurboModuleRegistry.get('ModuleName')`.

Lazy loading keeps startup time low: a module that handles background sync is never instantiated for a session where the user only reads cached content.

### Forcing Eager Initialization

To force a module to initialize before any JS call, implement `requiresMainQueueSetup` and return `YES`:

```objc
+ (BOOL)requiresMainQueueSetup {
  return YES;  // Instantiated on the main queue before the first JS frame
}
```

This is appropriate for modules that must configure global state (such as a logging SDK) before any code runs. Modules that return `YES` are instantiated synchronously during bridge or host startup, which adds to the startup cost — use sparingly.

### RCTTurboModuleSharedNothing

The `RCTTurboModuleSharedNothing` protocol marks a module as having no shared mutable state between instances. When adopted, the runtime is free to create per-caller instances rather than a singleton, which enables stricter isolation in test environments and allows the runtime to defer allocation further:

```objc
@interface RCTNativePayment : NSObject <RCTBridgeModule, RCTTurboModule, RCTTurboModuleSharedNothing>
@end
```

In production apps this protocol is rarely needed, but it is useful when writing modules that must behave identically whether called from the main surface or a secondary `RCTFabricSurface`.

## 6. Async TurboModules: RCTTurboModuleWithJSIBindings

Some modules need to install custom functions or objects directly into the JS runtime rather than exposing methods through the Codegen protocol. Examples include crash reporters that intercept JS exceptions, custom `console` overrides, and performance tracing tools.

These modules adopt `RCTTurboModuleWithJSIBindings`:

```objc
// RCTPerformanceTracer.h
@interface RCTPerformanceTracer : NSObject
    <RCTBridgeModule, RCTTurboModule, RCTTurboModuleWithJSIBindings>
@end

// RCTPerformanceTracer.mm
RCT_EXPORT_MODULE(NativePerformanceTracer)

- (void)installJSIBindingsWithRuntime:(facebook::jsi::Runtime &)runtime {
  using namespace facebook::jsi;

  // Install a __perfMark global function callable from JS
  auto markFn = Function::createFromHostFunction(
    runtime,
    PropNameID::forAscii(runtime, "__perfMark"),
    1,
    [](Runtime &rt, const Value &, const Value *args, size_t count) -> Value {
      if (count > 0 && args[0].isString()) {
        std::string label = args[0].getString(rt).utf8(rt);
        // Record the mark in native trace buffer
        PerformanceTraceBuffer::shared().mark(label);
      }
      return Value::undefined();
    }
  );
  runtime.global().setProperty(runtime, "__perfMark", std::move(markFn));
}

- (std::shared_ptr<facebook::react::TurboModule>)
    getTurboModule:(const facebook::react::ObjCTurboModule::InitParams &)params {
  return std::make_shared<facebook::react::NativePerformanceTracerSpecJSI>(params);
}
```

`installJSIBindingsWithRuntime:` is called once after the JS runtime is initialized, before the JS bundle executes. The function you install is available to the bundle immediately.

## 7. Synchronous TurboModules

Methods on a TurboModule that return a value directly (not via Promise) execute synchronously on the JS thread. There is no async queue, no serialization — JSI calls into the C++ dispatch table and the return value flows back to JS in the same stack frame.

```ts
// In the spec: synchronous return type
getLastTransactionId(): string | null;
```

```objc
// In the .mm: ObjC method called synchronously from JS
- (NSString *)getLastTransactionId {
  return [_helper getLastTransactionId];
}
```

For modules that need to return a fully custom `jsi::HostObject` rather than one of the Codegen-supported return types, override `getTurboModule:jsInvoker:` in the AppDelegate delegate to return the C++ object directly:

```objc
- (std::shared_ptr<facebook::react::TurboModule>)
    getTurboModule:(const std::string &)name
         jsInvoker:(std::shared_ptr<facebook::react::CallInvoker>)jsInvoker {
  if (name == "NativeCustomHost") {
    return std::make_shared<CustomHostObject>(jsInvoker);
  }
  return nullptr;
}
```

`CustomHostObject` extends `facebook::jsi::HostObject` and overrides `get` and `set` to intercept any property access from JS. This pattern is used by JSI-based libraries such as MMKV and Reanimated.

## 8. Testing

### Unit Testing the Swift Implementation with XCTest

The Swift helper class can be tested independently of React Native. No JSI, no ObjC++ glue, no Codegen required:

```swift
// PaymentHelperTests.swift
import XCTest
@testable import MyApp

final class PaymentHelperTests: XCTestCase {
  var helper: PaymentHelper!

  override func setUp() {
    super.setUp()
    helper = PaymentHelper()
  }

  func testGetLastTransactionIdReturnsNilInitially() {
    XCTAssertNil(helper.getLastTransactionId())
  }

  func testCancelPendingDoesNotThrow() {
    // Should be a no-op when nothing is pending
    XCTAssertNoThrow(helper.cancelPending())
  }

  func testChargeStoresTransactionId() async throws {
    // Using a URLProtocol stub for network isolation
    URLProtocol.registerClass(MockPaymentProtocol.self)
    defer { URLProtocol.unregisterClass(MockPaymentProtocol.self) }

    MockPaymentProtocol.responseData = #"{"transactionId":"tx_abc123"}"#.data(using: .utf8)

    let txId = try await helper.charge(amount: 9.99, currency: "USD")

    XCTAssertEqual(txId, "tx_abc123")
    XCTAssertEqual(helper.getLastTransactionId(), "tx_abc123")
  }
}
```

This is the primary testing surface for iOS TurboModules — business logic belongs in Swift, where XCTest can reach it without native runtime setup.

### Mocking the JS Side with Jest

On the JavaScript side, `TurboModuleRegistry.strictGet` fails in Jest because there is no native runtime. Mock the spec module before any import that uses it:

```ts
// __mocks__/NativePayment.ts  (place next to the spec file)
const NativePaymentMock = {
  charge: jest.fn().mockResolvedValue('tx_test_001'),
  getLastTransactionId: jest.fn().mockReturnValue('tx_test_001'),
  cancelPending: jest.fn(),
};

export default NativePaymentMock;
```

In your test:

```ts
// payment.test.ts
jest.mock('../specs/NativePayment');
import NativePayment from '../specs/NativePayment';
import { processPayment } from '../services/PaymentService';

describe('PaymentService', () => {
  beforeEach(() => jest.clearAllMocks());

  it('calls charge with correct arguments', async () => {
    await processPayment(19.99, 'BRL');
    expect(NativePayment.charge).toHaveBeenCalledWith(19.99, 'BRL');
  });

  it('returns the transaction id', async () => {
    const result = await processPayment(19.99, 'BRL');
    expect(result.transactionId).toBe('tx_test_001');
  });
});
```

## 9. Migration from Legacy NativeModules

Modules using the legacy `RCTBridgeModule` protocol continue to work in RN 0.76+ through the TurboModule interop layer, which wraps them in a TurboModule facade automatically. You can migrate incrementally without breaking the app.

### Side-by-Side During Migration

Keep the legacy `RCT_EXPORT_MODULE` and `RCT_EXPORT_METHOD` declarations while adding the TurboModule conformance:

```objc
// RCTNativePayment.mm — during migration
@interface RCTNativePayment : NSObject
    <RCTBridgeModule,    // legacy — keeps existing callers working
     RCTTurboModule,     // new — enables TurboModule resolution
     NativePaymentSpec>  // new — Codegen-generated protocol
@end

@implementation RCTNativePayment

RCT_EXPORT_MODULE(NativePayment)

// Legacy export — used by the interop layer until the spec is available
RCT_EXPORT_METHOD(charge:(double)amount
                  currency:(NSString *)currency
                  resolver:(RCTPromiseResolveBlock)resolve
                  rejecter:(RCTPromiseRejectBlock)reject) {
  [self charge:amount currency:currency resolve:resolve reject:reject];
}

// New TurboModule method — used by JS once the spec is wired up
- (void)charge:(double)amount
      currency:(NSString *)currency
        resolve:(RCTPromiseResolveBlock)resolve
         reject:(RCTPromiseRejectBlock)reject {
  // shared implementation
}

- (std::shared_ptr<facebook::react::TurboModule>)
    getTurboModule:(const facebook::react::ObjCTurboModule::InitParams &)params {
  return std::make_shared<facebook::react::NativePaymentSpecJSI>(params);
}

@end
```

Once all call sites use the typed spec and the TypeScript layer no longer uses `NativeModules.NativePayment`, remove the `RCT_EXPORT_METHOD` declarations and the legacy `RCTBridgeModule` conformance.

### Interop Layer Caveats

The interop layer covers most method types but has known gaps: callbacks (not Promises), methods that accept `RCTResponseSenderBlock`, and modules that override `methodQueue` to run on a custom serial queue may behave differently under the interop layer. Test each migrated module individually before removing the legacy export.

## 10. Common Pitfalls

### Forgetting RCT_EXPORT_MODULE

Without `RCT_EXPORT_MODULE(NativePayment)`, the Objective-C runtime never registers the class with React Native's module registry. The `TurboModuleManager` delegate's `getModuleClassFromName:` will not find it, and JavaScript receives a "module not found" error at runtime, not at build time.

The macro must appear inside `@implementation`, not inside `@interface`. Placing it in the header causes a linker error.

### Module Name Mismatch Between Spec and .mm

The string in `RCT_EXPORT_MODULE(NativePayment)` must match the string in `TurboModuleRegistry.strictGet('NativePayment')` exactly, including case. A common mistake is using `RCT_EXPORT_MODULE()` with no argument, which defaults to the class name (e.g., `RCTNativePayment`) — which then does not match `'NativePayment'` in the spec.

Always pass the module name explicitly:

```objc
RCT_EXPORT_MODULE(NativePayment)   // matches TurboModuleRegistry.strictGet('NativePayment')
```

### Pod Not Added to Podfile

If the module is distributed as a separate pod and not listed in the consuming app's `Podfile`, Codegen never runs for it, the generated headers are absent, and the Xcode build fails with "file not found" on the import of the generated spec header. Every pod that contains a TurboModule must appear in `Podfile` before `pod install`:

```ruby
# Podfile
pod 'NativePaymentModule', path: '../modules/native-payment'
```

Re-run `bundle exec pod install` after adding the entry.

### Missing pod_target_xcconfig for C++ Standard

The generated `.mm` files use C++17 features. Without the correct compiler flag the build fails with obscure template errors:

```ruby
# In your podspec:
s.pod_target_xcconfig = {
  "CLANG_CXX_LANGUAGE_STANDARD" => "c++17",
  "HEADER_SEARCH_PATHS" => "$(PODS_ROOT)/boost"
}
```

### Wrong Codegen Name in package.json

The `name` field in `codegenConfig` becomes the iOS framework name and the prefix for generated files. If this name does not match the import in your `.mm` file, the build fails:

```objc
// If codegenConfig.name is "AppSpecs":
#import <AppSpecs/AppSpecsJSI.h>  // correct

// If codegenConfig.name is "MyModuleSpecs":
#import <MyModuleSpecs/MyModuleSpecsJSI.h>  // must match
```

Run `pod install` and inspect `build/generated/ios/` to confirm the actual directory name before writing the import.

### getTurboModule: Returning a Null Pointer

If `getTurboModule:` returns a null `shared_ptr`, the runtime silently falls back to the legacy module resolution path (or returns undefined to JS). Add a log statement inside the method during development to confirm it is being reached:

```objc
- (std::shared_ptr<facebook::react::TurboModule>)
    getTurboModule:(const facebook::react::ObjCTurboModule::InitParams &)params {
  NSLog(@"[RCTNativePayment] getTurboModule: called");
  return std::make_shared<facebook::react::NativePaymentSpecJSI>(params);
}
```

If the log never appears, the class is not being resolved by the delegate — check that `RCT_EXPORT_MODULE` is present and the module name matches.
