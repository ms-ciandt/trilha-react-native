# Process: Upgrade Strategy

> Upgrading React Native is not a single command. It is a structured migration: diff, apply, validate, repeat. The developers who get it right treat it like a planned refactor — not a dependency bump.

---

## Why Incremental Upgrades Win

React Native releases a new minor version roughly every 6 weeks. Each minor carries changes in three layers simultaneously: the JS APIs, the native Android build system (Gradle plugin, CMake), and the native iOS setup (Podfile, AppDelegate, Xcode project settings).

Jumping multiple minors at once makes it impossible to isolate which change broke what. The correct strategy is **one minor at a time**, with a full build validation between each step.

```
0.73 ──validate──► 0.74 ──validate──► 0.75 ──validate──► 0.76
```

Not:

```
0.73 ──────────────────────────────────────────────────► 0.76  ← 3x harder to debug
```

For patch upgrades (`0.76.0 → 0.76.5`), it is safe to jump all patches at once — patches are bug-fixes and security fixes with no intentional breaking changes.

---

## The Four-Phase Upgrade Process

```
Phase 1: Analyse     → read changelog, check library compatibility
Phase 2: Diff        → get native diffs from Upgrade Helper
Phase 3: Apply       → bump packages, apply native diffs, recreate patches
Phase 4: Validate    → build on both platforms, run tests, smoke test on device
```

Do not skip Phase 1. The most expensive mistakes come from applying diffs before understanding what changed semantically.

---

## Phase 1: Read the Changelog Before Touching Code

```bash
# Open the official changelog for your target range
open https://github.com/facebook/react-native/blob/main/CHANGELOG.md
```

Look specifically for:
- **Breaking Changes** section — anything that requires code changes
- **Deprecated** — APIs being removed in the next version
- **Android** / **iOS** subsections — native-only changes that don't appear in JS diffs
- Any mention of your own native modules or heavily used third-party libraries

Also read the reactwg/react-native-releases discussion for your target version — it surfaces known issues found during the RC cycle before the final release notes are written.

```bash
# Example: discussion thread for 0.76
open https://github.com/reactwg/react-native-releases/discussions
```

## Phase 2: RN Upgrade Helper & Native Diffs

The **Upgrade Helper** generates a clean diff between any two RN versions by running `react-native init` at both versions and diffing the output.

**URL:** [react-native-community.github.io/upgrade-helper](https://react-native-community.github.io/upgrade-helper/)

How to use it:

1. Set "From" to your current version (e.g., `0.75.4`)
2. Set "To" to your target version (e.g., `0.76.7`)
3. Click "Show me how to upgrade"
4. Work through the diff **file by file**, top to bottom

### The files that matter most

| File | What changes | Risk level |
|---|---|---|
| `package.json` | RN version, React version, peer deps | Low — straightforward version bumps |
| `android/build.gradle` | Gradle plugin version, classpath, repositories | Medium — wrong version = build failure |
| `android/app/build.gradle` | compileSdk, targetSdk, minSdk, build tools | High — SDK changes affect behavior |
| `android/gradle/wrapper/gradle-wrapper.properties` | Gradle wrapper version | Medium — must match the plugin |
| `android/app/src/main/AndroidManifest.xml` | permissions, activity flags, themes | High — missing flags = runtime crash |
| `ios/Podfile` | platform version, `use_react_native!` options, flags | High — wrong options = link failures |
| `ios/AppDelegate.swift` / `.mm` | RCTHost setup, lifecycle methods | High — wrong delegate = app fails to boot |
| `ios/Podfile.lock` | Re-generated by pod install | Low — do not edit manually |

### Reading the diff for native-specific changes

The Upgrade Helper shows exactly which lines changed in each file. For a `build.gradle` diff like:

```diff
- compileSdkVersion = 33
+ compileSdkVersion = 35

- targetSdkVersion = 33
+ targetSdkVersion = 35

- minSdkVersion = 21
+ minSdkVersion = 24
```

This is not just a number bump — `targetSdkVersion = 35` triggers Android 15's edge-to-edge enforcement (see the Native Settings topic). `minSdkVersion = 24` drops support for Android 7.0 devices.

### Using the diff in a brownfield app

In a brownfield app, your `AppDelegate`, `MainApplication`, and Gradle files are not generated from a template — they are hand-authored. The Upgrade Helper diff shows template changes; you must **map** those changes onto your custom files.

```bash
# Recommended: keep a side-by-side terminal
# Left pane: Upgrade Helper showing the diff
# Right pane: your actual AppDelegate.swift / build.gradle
```

For Gradle, apply changes at the property level — do not copy-paste the entire generated file over your brownfield file. One wrong overwrite erases custom configurations you may not have documented.

## Phase 3: Applying the Upgrade

```bash
# 1. Bump JS packages
npm install react-native@0.76.7 react@18.3.1

# Or with Yarn
yarn add react-native@0.76.7 react@18.3.1

# 2. Run the automated file updater (bare workflow)
npx react-native upgrade

# This command uses rn-diff-purge internally and attempts
# to apply the native file changes automatically.
# It will mark conflicts (<<<< HEAD) where it cannot merge.
```

> `npx react-native upgrade` is a best-effort tool — it works well on standard template projects but often leaves conflicts on brownfield or heavily customized projects. In those cases, apply changes manually using the Upgrade Helper diff as your guide.

After applying JS changes:

```bash
# iOS: reinstall pods with the new versions
cd ios && bundle exec pod install && cd ..

# Android: clean Gradle caches (do this on every upgrade)
cd android && ./gradlew clean && cd ..
```

## Phase 4: Breaking Change Analysis Checklist

Before building, audit against the changelog:

```
[ ] JS API changes applied (deprecated props, renamed APIs)
[ ] Native module interfaces match new generated specs (if you have custom TurboModules)
[ ] AndroidManifest changes applied
[ ] AppDelegate / SceneDelegate changes applied (iOS)
[ ] Podfile options updated (new flags like new_arch_enabled, fabric_enabled)
[ ] Gradle plugin version matches wrapper version
[ ] targetSdk implications reviewed (edge-to-edge if 35)
[ ] Third-party library compatibility verified (react-native.directory)
[ ] patches/ dir reviewed — any patches touching upgraded files?
[ ] Tests still pass: yarn test
```

## Recommended Upgrade Path to 0.76+ (New Architecture Default)

If you are still on 0.72 or earlier, this is the recommended sequence:

```
0.72 → 0.73   (JSC → Hermes default; Gradle plugin v8)
0.73 → 0.74   (Bridgeless preview; minSdk 23; Kotlin 1.9)
0.74 → 0.75   (minSdk 24; Xcode 15.1 required; Swift AppDelegate)
0.75 → 0.76   (New Architecture ON by default; ReactHost; edge-to-edge prep)
0.76 → 0.77+  (each minor: stable under the new arch)
```

Key reason to not skip 0.74: it introduced the **TurboModule Interop Layer** in a stable form. Without going through 0.74, you may hit interop issues when you land on 0.76 with New Architecture on.

## Validation: Build on Both Platforms

**Always build from a clean state after an upgrade.** Cached build artifacts from the previous version cause non-deterministic failures.

```bash
# Full clean build — Android
cd android
./gradlew clean
./gradlew assembleRelease

# Full clean build — iOS
cd ios
rm -rf build/
pod install
xcodebuild -workspace MyApp.xcworkspace \
           -scheme MyApp \
           -configuration Release \
           -sdk iphoneos \
           clean build
```

**Smoke test checklist on device (not simulator):**

```
[ ] App launches to home screen
[ ] Navigation between main screens works
[ ] Network calls succeed (auth, data fetch)
[ ] Any custom native module: basic call succeeds
[ ] Camera / biometrics / push notifications (if used) still work
[ ] No JS exceptions in Metro logs on startup
[ ] Performance: no obvious jank on scroll-heavy screens
```

Run on a **low-end device** for the Android smoke test — most upgrade-introduced performance regressions are invisible on flagship hardware.
