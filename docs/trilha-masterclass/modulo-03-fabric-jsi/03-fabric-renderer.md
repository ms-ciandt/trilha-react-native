## 1. Why a New Renderer

The old renderer (the "paper" renderer) had one structural problem: it was written in Objective-C and Java, which meant the C++ core could not reason about views without going through a language boundary. Every layout event involved:

1. JS thread calls layout (via bridge)
2. Shadow thread computes Yoga layout in C++
3. Result serialized back to Obj-C/Java
4. Obj-C/Java creates/updates native views

The layout information existed as a copy in four places simultaneously. Mutation of that copy required locks, and the locks meant the UI thread could block the JS thread and vice versa.

Fabric solves this by moving **the entire view description into C++**. Native views become thin facades. The real data model lives in C++ and is accessible directly from the JS thread via JSI — no serialization.

The result: layout is synchronous, view mutations are atomic, and the renderer can participate in React's concurrent rendering model (which requires the ability to interrupt and resume work without leaving the view tree in an inconsistent state).

## 2. Shadow Tree and Reconciliation

### What a Shadow Tree is

The Shadow Tree is the C++ representation of the React component tree. Every `<View>`, `<Text>`, `<Image>` you write in JSX has a corresponding **Shadow Node** in C++ that:
- Holds the component's props as a typed C++ struct
- Holds the Yoga layout node
- Holds the computed layout metrics (position, size)

Shadow Nodes are **immutable once committed**. You never mutate a committed node — you clone it, apply changes, and commit the new tree. This is the same philosophy as React's virtual DOM — but implemented in C++.

```
React Tree (JS)                Shadow Tree (C++)            Mounted Tree (Native)
─────────────────              ──────────────────           ──────────────────────
<View style={{flex:1}}>   →   ViewShadowNode               UIView / android.view.View
  <Text>Hello</Text>       →     TextShadowNode             UITextView / TextView
  <Image src={...} />      →     ImageShadowNode            UIImageView / ImageView
</View>
```

### Shadow Node structure

```cpp
// react/renderer/components/view/ViewShadowNode.h (simplified)
class ViewShadowNode final : public ConcreteViewShadowNode<
    ViewComponentName,    // "RCTView"
    ViewShadowNodeFragment,
    ViewEventEmitter,
    ViewProps            // typed C++ struct derived from JS props
> {
public:
  using ConcreteViewShadowNode::ConcreteViewShadowNode;
};
```

`ViewProps` is a C++ struct that Codegen produces from the component's TypeScript spec. Every prop has a typed field — no `NSDictionary` / `ReadableMap` lookups at runtime.

```cpp
// ViewProps.h — generated by Codegen
class ViewProps : public YogaStylableProps, public AccessibilityProps {
public:
  // Each prop has a C++ field with its default value
  SharedColor backgroundColor{};
  Float opacity{1.0};
  EdgeInsets borderWidth{};
  BorderStyle borderStyle{BorderStyle::Solid};
  // ...
};
```

### Reconciliation: three trees in flight

Fabric maintains three trees simultaneously:

| Tree | Description |
|---|---|
| **Current** | The committed tree currently on screen |
| **Work-in-progress** | The next tree being built from the new React state |
| **Rendering** | The tree being transferred to native views (may differ from both above in concurrent mode) |

The reconciler receives a new React element tree from the JS reconciler (React's virtual DOM differ), creates a new Shadow Tree from it, computes Yoga layout, and commits — all in C++.

### The commit pipeline

```
React (JS) calls setState
        │
        ▼
React reconciler produces new element tree
        │
        ▼
Fabric C++ receives element tree via JSI
        │
        ▼
ShadowTreeCommitter::commit()
  ├─ Clone changed ShadowNodes (immutable — never mutate in place)
  ├─ Apply new props
  ├─ Perform Yoga layout pass (calculateLayout)
  └─ Produce LayoutMetrics for each node
        │
        ▼
MountingCoordinator schedules mutations
        │
        ▼
UIThread applies mutations to native views
  ├─ Create new views
  ├─ Update existing views (props, layout)
  └─ Delete removed views
```

### Yoga layout engine

Yoga is the C++ flexbox implementation that powers layout in both Fabric and the old renderer. Its public interface is straightforward:

```cpp
#include <yoga/Yoga.h>

// Create a node for each component
YGNodeRef root = YGNodeNew();
YGNodeStyleSetFlexDirection(root, YGFlexDirectionColumn);
YGNodeStyleSetWidth(root, 375);
YGNodeStyleSetHeight(root, 812);

YGNodeRef child = YGNodeNew();
YGNodeStyleSetFlex(child, 1);
YGNodeInsertChild(root, child, 0);

// Calculate layout (call once per commit)
YGNodeCalculateLayout(root, YGUndefined, YGUndefined, YGDirectionLTR);

// Read results — these are the pixel values for native views
float childLeft   = YGNodeLayoutGetLeft(child);   // e.g. 0
float childTop    = YGNodeLayoutGetTop(child);    // e.g. 0
float childWidth  = YGNodeLayoutGetWidth(child);  // e.g. 375
float childHeight = YGNodeLayoutGetHeight(child); // e.g. 812

YGNodeFreeRecursive(root);
```

In Fabric, Yoga nodes are embedded inside ShadowNodes. You do not call Yoga directly — the ShadowTree manages it. But understanding Yoga is essential because every `style` prop on a React Native component maps to a `YGNodeStyle*` call.

**Flexbox playground for Yoga mental model:** https://yogalayout.dev/playground

## 3. Concurrent Rendering

### What Concurrent Mode means for Fabric

React 18's concurrent features (Suspense, `useTransition`, `useDeferredValue`, automatic batching) require the renderer to be interruptible. The old Fabric (RN 0.68–0.71) started the concurrent foundation; RN 0.76 ships with full concurrent rendering enabled by default.

What "interruptible" means in practice:

```typescript
// With useTransition, React can de-prioritize this update
const [isPending, startTransition] = useTransition();

function handleSearch(query: string) {
  // This update can be interrupted if a higher-priority update arrives
  startTransition(() => {
    setSearchQuery(query);
  });
}
```

When `startTransition` is active and the user triggers another event (scroll, tap), React can **discard the in-progress render** of the search results and prioritize the user interaction. Without concurrent rendering, the render was synchronous and could not be interrupted — causing dropped frames.

For Fabric, this means the work-in-progress Shadow Tree can be discarded before commit. Fabric's immutable node design makes this safe: nothing was mutated, so nothing needs to be "undone".

### React update priorities in Fabric

Fabric maps React's update priorities (from `react-reconciler`) to Fabric commit priorities:

| React Priority | Fabric Priority | Use case |
|---|---|---|
| `DiscreteEventPriority` | Synchronous | Tap, press — must not delay |
| `ContinuousEventPriority` | Synchronous | Scroll, drag |
| `DefaultEventPriority` | Asynchronous | State updates from data fetch |
| `IdlePriority` | Deferred | Offscreen rendering |

In RN 0.76, discrete events (taps) still commit synchronously — if you tap a button and the handler calls `setState`, the new state reflects in the same frame. But a `startTransition` update can be split across frames.

### `useTransition` — practical example

```typescript
import React, { useState, useTransition, Suspense } from 'react';
import { FlatList, TextInput, ActivityIndicator, View } from 'react-native';

// A deliberately expensive filter over a large list
function filterProducts(query: string, products: Product[]): Product[] {
  return products.filter(p =>
    p.name.toLowerCase().includes(query.toLowerCase()) ||
    p.description.toLowerCase().includes(query.toLowerCase())
  );
}

export function ProductSearch({ products }: { products: Product[] }) {
  const [query, setQuery] = useState('');
  const [filtered, setFiltered] = useState(products);
  const [isPending, startTransition] = useTransition();

  const handleChange = (text: string) => {
    setQuery(text);  // high priority — TextInput stays responsive

    startTransition(() => {
      // low priority — this can be interrupted if user types again
      setFiltered(filterProducts(text, products));
    });
  };

  return (
    <View style={{ flex: 1 }}>
      <TextInput value={query} onChangeText={handleChange} />
      {isPending && <ActivityIndicator />}
      <FlatList
        data={filtered}
        keyExtractor={p => p.id}
        renderItem={({ item }) => <ProductRow product={item} />}
      />
    </View>
  );
}
```

Here, `setQuery` always commits synchronously — the `TextInput` value never lags. The `setFiltered` inside `startTransition` is interruptible — if you type the next character before the filter finishes, React throws away the previous in-progress render.

## 4. Fabric Threading Model

### The three threads

```
┌─────────────────────────────────────────────────────────────────┐
│  JS Thread                                                       │
│  - Runs Hermes VM                                                │
│  - Executes React reconciler                                     │
│  - Builds React element trees                                    │
│  - Calls Fabric via JSI (no queue, no serialization)            │
└──────────────────────────────┬──────────────────────────────────┘
                               │ JSI (direct C++ call)
┌──────────────────────────────▼──────────────────────────────────┐
│  Background Thread (Fabric Commit Thread)                        │
│  - Clones ShadowNodes                                            │
│  - Runs Yoga layout calculation                                  │
│  - Produces MountingTransactions (lists of view mutations)       │
└──────────────────────────────┬──────────────────────────────────┘
                               │ MountingTransaction (atomic batch)
┌──────────────────────────────▼──────────────────────────────────┐
│  UI Thread (Main Thread)                                         │
│  - Creates / updates / deletes native views                      │
│  - Runs platform animations                                      │
│  - Processes touch events                                        │
└─────────────────────────────────────────────────────────────────┘
```

### What changed from the old architecture

The old Shadow Thread did layout but could not talk to the JS thread without going through the bridge. Fabric eliminates the bridge:

- The JS thread calls Fabric's C++ renderer **directly via JSI** — no serialization, no queue
- The commit thread is a background thread that Fabric manages internally
- The UI thread applies mutations delivered as a `MountingTransaction` — an atomic list of `Create`, `Update`, `Delete`, `LayoutUpdate` instructions

### Synchronous commits on the UI thread (Synchronous Rendering)

When an event is `DiscreteEventPriority` (a tap), Fabric can perform the entire pipeline synchronously on the UI thread:

```
UI Thread receives tap
  └─► dispatch event to JS
        └─► React re-renders synchronously
              └─► Fabric commits synchronously on UI thread
                    └─► Native views updated in same frame
```

This is why pressing a `<Pressable>` in RN 0.76 feels instant — there is no thread hop. The entire update completes on the same thread that received the touch event.

### Thread safety in Fabric components

If you write a native Fabric component, your C++ methods are called from different threads:

```cpp
class MyFabricComponentShadowNode final
    : public ConcreteViewShadowNode<...> {
public:
  // Called on the Fabric commit/background thread
  void layout(LayoutContext layoutContext) override {
    // Safe: only reads props, never touches native views
    auto size = getLayoutMetrics().frame.size;
  }
};

// Your native view class
class MyFabricView : public RCTViewComponentView {  // iOS
public:
  // Called on the UI thread
  void updateProps(Props::Shared const& props, Props::Shared const& oldProps) override {
    // Safe: runs on UI thread, can mutate UIKit views
    auto& concreteProps = *std::static_pointer_cast<MyFabricComponentProps const>(props);
    self.label.text = RCTNSStringFromString(concreteProps.title);
  }
};
```

The thread boundary is enforced by the framework: ShadowNode methods run on the background thread, View methods run on the UI thread. Do not cross this boundary.
